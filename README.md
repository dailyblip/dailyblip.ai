# dailyblip

The AI brief for people who make things. A static site + serverless-free pipeline: GitHub Actions fetch creator-focused AI news every 20 minutes, Claude classifies and summarizes it, and once a day it writes the 60-second Daily Blip, picks a Tool Drop, and sends the newsletter.

## How it fits together

```
RSS sources ──▶ ingest.js (every 20 min) ──▶ docs/data/feed.json ──▶ static site (GitHub Pages)
                     │                              ▲
                     ▼                              │
              Claude (Haiku) classifies      brief.js + tooldrop.js (daily, Claude Sonnet)
              & writes deks                         │
                                                    ▼
                                             newsletter.js ──▶ Buttondown ──▶ inboxes
```

There is no server. The "database" is `feed.json`, committed to the repo by the bot. The site is pure static files that hydrate from that JSON (and fall back to embedded sample data, so it also works opened straight from disk).

```
dailyblip/
├── docs/                  # the website — GitHub Pages serves this folder
│   ├── index.html
│   └── data/feed.json     # generated by the pipeline (seeded with sample data)
├── docs/archive/          # immutable daily snapshots (permalinks + JSON) — compounds forever
├── pipeline/
│   ├── ingest.js          # RSS → dedupe → ground in article text → Claude classify/summarize (+ Spotlight flag)
│   ├── brief.js           # daily: rank top 6, write the Daily Blip (publish-gated, no-repeat memory)
│   ├── tooldrop.js        # daily: pick + verify the Tool Drop of the Day
│   ├── archive.js         # daily: write the day's permalink page + "one year ago" panel
│   ├── curator.js         # weekly: self-heal dead sources (Claude + web search), tool graveyard check
│   ├── newsletter.js      # daily: send the brief via Buttondown
│   ├── sources.json       # feed list — the curator maintains this itself
│   └── lib/               # claude.js (API + model fallback), store.js (IO), text.js (dedupe), sanitize.js (gates)
├── data/
│   ├── seen.json          # URLs already judged (pruned to 14 days)
│   ├── featured.json      # tool-drop history + graveyard (alive/died dates)
│   ├── health.json        # per-source item counts — the curator's vital signs
│   └── models.json        # written by the model-fallback logic when a model is retired
└── .github/workflows/
    ├── ingest.yml         # cron: every 20 min
    ├── daily.yml          # cron: 6am PT — brief, tool drop, archive, newsletter
    └── curator.yml        # cron: weekly — source healing + graveyard
```

## Step-by-step setup

### Step 1 — Local sanity check (5 min)

```bash
npm install
npm run serve          # → http://localhost:3000, site renders from the seeded feed.json
```

You should see the full site with sample data. Nothing else is required to preview.

### Step 2 — Get an Anthropic API key (5 min)

1. Create an account at https://console.anthropic.com and add a payment method.
2. Create an API key under **API Keys**.
3. Test the pipeline locally:

```bash
export ANTHROPIC_API_KEY=sk-ant-...
npm run ingest         # fetches RSS, classifies with Claude, updates docs/data/feed.json
npm run brief          # writes the Daily Blip from the last 24h of stories
npm run tooldrop       # picks and verifies today's tool drop
```

Open `docs/data/feed.json` after each run to see what changed. API docs, models, and pricing: https://docs.claude.com/en/api/overview. The pipeline uses **Claude Haiku 4.5** for high-volume classification and **Claude Sonnet 4.6** for the editorial writing — at this volume (one small batched call per ingest run, two calls per day for brief + tool drop) expect on the order of a few dollars per month, but verify against current pricing on the docs.

### Step 3 — Push to GitHub and turn on Pages (10 min)

```bash
git init && git add -A && git commit -m "dailyblip v1"
git remote add origin git@github.com:YOURNAME/dailyblip.git
git push -u origin main
```

Then in the repo on github.com:

1. **Settings → Pages** → Source: *Deploy from a branch* → Branch: `main`, folder: `/docs` → Save.
2. Wait ~1 minute; your site is live at `https://YOURNAME.github.io/dailyblip/`.

### Step 4 — Wire up the automation (10 min)

1. **Settings → Secrets and variables → Actions → New repository secret**:
   - `ANTHROPIC_API_KEY` = your key from Step 2.
2. **Settings → Actions → General**: under *Workflow permissions*, select **Read and write permissions** (the bot commits `feed.json`).
3. Go to the **Actions** tab → *Ingest feed* → **Run workflow** to test manually. A green run should end with a new commit like `feed: 2026-07-08 17:40 UTC`, and the live site updates on the next Pages deploy (~1 min).
4. Run *Daily Blip* manually once too — it writes the brief and tool drop.

From here everything is automatic: ingest every 20 minutes, the Blip at 6am PT. (Note: GitHub cron is best-effort and can run a few minutes late; that's fine for this use.)

### Step 5 — The custom domain (15 min + DNS wait)

1. Register `dailyblip.ai` (.ai domains run ~$70–90/yr; any registrar works).
2. Repo **Settings → Pages → Custom domain** → enter `dailyblip.ai`. GitHub creates a `CNAME` file in `/docs` — commit it if prompted.
3. At your registrar, add DNS records:
   - `A` records for the apex → `185.199.108.153`, `.109.153`, `.110.153`, `.111.153`
   - `CNAME` for `www` → `YOURNAME.github.io`
4. Back in Pages settings, tick **Enforce HTTPS** once the DNS check passes (can take up to a day).

(Alternative: Cloudflare Pages or Vercel also serve `/docs` fine and deploy faster; GitHub Pages is used here because it requires zero extra accounts.)

### Step 6 — The newsletter (20 min)

1. Create a free account at https://buttondown.com and note your **username** and **API key** (Settings → API).
2. In `docs/index.html`, find `<form class="row" id="subForm" data-buttondown="">` and set `data-buttondown="YOUR_USERNAME"` — the subscribe box now adds real subscribers.
3. Add a second repo secret: `BUTTONDOWN_API_KEY`.
4. That's it — `newsletter.js` already runs in the daily workflow and skips silently until the key exists. Verify the exact send-API fields against https://docs.buttondown.com before your first real send, and send yourself a test by triggering *Daily Blip* manually.

### Step 7 — Tune the editorial voice (ongoing)

- **Sources**: edit `pipeline/sources.json`. Each feed takes a `hint` category; the classifier can override it. Reddit feeds are noisy — the classifier prompt drops non-news, but prune any feed that keeps slipping junk through.
- **Selection taste**: the rubrics live as system prompts at the top of `ingest.js`, `brief.js`, and `tooldrop.js`. This is where the product actually lives — if the brief feels off, edit the ranking rules ("deadlines outrank launches…") rather than the code.
- **Manual override**: to pin tomorrow's tool drop, just edit `feed.tooldrop` in `docs/data/feed.json` and add the name to `data/featured.json` so the job doesn't overwrite it with a repeat.
- **Sponsorship policy**: if you ever sell the tool-drop slot, label it in the UI. An unlabeled paid slot destroys the feature's credibility.

## Built for unattended, multi-year operation

This system is designed to run without a human. The specific mechanisms:

- **Self-healing sources.** Every ingest run records per-feed item counts into `data/health.json`. The weekly curator flags any feed that's been silent for 7 days or failing for 5 straight runs, then uses Claude *with web search* to find the feed's new URL or a replacement covering the same beat. Proposed replacements are verified (must actually parse with items) before `sources.json` is edited; unverifiable ones are disabled with a reason, never guessed at.
- **Model-proof API calls.** If a pinned Claude model is ever retired, `lib/claude.js` catches the error, queries the Models API for the newest model in the same tier (haiku for classification, sonnet for writing), persists the switch to `data/models.json`, and retries. Model deprecations never take the pipeline down.
- **Publish gates.** The brief must pass validation (4–6 items, valid story IDs, no duplicates, HTML sanitized to `<b>` only) or yesterday's brief stays live. Deks are grounded in fetched article text — the classifier is instructed to omit anything the excerpt doesn't support. The tool drop never ships an unverified URL. The failure posture everywhere is *keep the last good state*.
- **Injection resistance.** Model-generated brief HTML is sanitized server-side at the gate AND client-side at render; a hostile RSS item can't script its way onto the page.
- **Dead man's switch.** If `feed.json` hasn't been regenerated in 48 hours, the site automatically shows a "feed paused — showing last known signals" banner rather than passing off stale news as fresh. Pair with GitHub's email-on-workflow-failure (Watch → Custom → Actions on your own repo) so you hear about breakage.
- **Compounding archive.** Every day the pipeline writes an immutable `docs/archive/YYYY-MM-DD` page + JSON and rebuilds the archive index. This gives you permalinks, an SEO surface that grows daily, and — starting in year two, automatically — a "one year ago today" sidebar panel. The curator also link-checks past tool drops (two-strike rule) and records death dates in `featured.json`: over years, the tool graveyard becomes data nobody else has.

## The Creator Spotlight

The classifier flags stories that are primarily about a *person or small team making something interesting with AI* — a filmmaker's short, an indie artist's AI-visual album, a novel workflow — as `spotlight: true` (capped at the 3 freshest). These render as an aqua-accented showcase rail between the brief and the feed, get a ✦ flag on their feed cards, and have their own filter chip in the nav. The brief may include at most one spotlight story, and only if it's genuinely remarkable. Showcase-heavy sources (80 Level, subreddit daily tops) exist in `sources.json` mainly to feed this. If the rail runs thin, add more maker-story sources; if it fills with product news, tighten the spotlight definition in the `ingest.js` prompt.

## The admin interface

Sometimes you need to remove a post, fix a typo, or add a story the pipeline missed. `docs/admin.html` is a self-contained editor for exactly that — no backend, no auth service, no CMS to run. It talks directly to your GitHub repo through the Contents API and every save becomes a real commit.

**One-time setup (2 min):**

1. Visit `https://dailyblip.ai/admin.html` (or bookmark it — it's `noindex, nofollow` so it won't appear in search results).
2. Create a fine-scoped personal access token at [github.com/settings/tokens](https://github.com/settings/tokens?type=beta):
   - Resource owner: your account
   - Repositories: **only** `dailyblip`
   - Permissions: **`Contents: Read and write`** (nothing else — this token can only edit files in this one repo)
3. Enter `username/dailyblip` and the token in the admin's connect dialog. The token is stored in this browser's localStorage — never sent anywhere except GitHub, and the `disconnect` button forgets it.

**What you can do:**

- **Stories tab** — search, filter (all / spotlight / locked / top), inline-edit any story's headline, dek, category, badge, source, URL, and read time. Per-story buttons: 🔒 lock (protects it from being overwritten by ingest), ✦ spotlight (adds it to the showcase rail), ◉ top (promotes it to top signal — exclusive across the feed), and delete (removes it AND adds its ID to the blocklist so ingest can never re-add it).
- **Today's brief** — edit the title and each of the 6 items directly; the dropdown re-points an item to a different story, seconds are editable, HTML is limited to `<b>` at render.
- **Tool drop** — hand-write today's drop. Marking anything here pins it: the daily tool-drop job sees `pinned: true` and skips picking. Hit *clear* when you want the pipeline to resume.
- **Overrides** — the standing-rules panel. Blocked story IDs (populated automatically by deletes), blocked terms (kill any story whose title contains e.g. "crypto"), and pinned spotlight IDs (force stories into the rail).
- **+ Add story** — for editorial additions the pipeline missed. Manual stories are automatically locked so they aren't touched, and can be flagged spotlight or top on creation.

**How edits are respected by the pipeline:**

- `locked: true` and `manual: true` stories are never overwritten by ingest.
- `data/overrides.json` (blocklist, term filter, pinned spotlights) is read by ingest before publishing.
- `feed.tooldrop.pinned: true` makes the tool-drop job leave that day's pick alone.
- Deleted stories go on the blocklist automatically — they can't come back through re-ingestion.

**Publishing:** the *Publish* button in the top bar commits `feed.json` and `overrides.json` in one shot, GitHub Pages redeploys in about a minute, and the live site updates. A dirty-state indicator lives in the top bar; the browser also warns before you leave with unsaved changes.

Works on phone. Every button is finger-sized. Reload-safe. If something breaks, `↻ reload` pulls the current committed state; nothing you did while offline is lost on this device until you hit Publish.

## Operating notes

- **Cost profile**: hosting $0, Actions $0 (public repo), Buttondown $0 to start, Claude API a few dollars/month, domain ~$80/yr.
- **Failure behavior**: every job fails safe (see the durability section above). Set a monthly spend cap in the Anthropic console so a runaway loop can never surprise you. Note: scheduled workflows on repos with no activity for 60 days get auto-disabled by GitHub — the bot's own commits count as activity, so a healthy pipeline keeps itself alive, but re-enable workflows if you ever pause it for months.
- **Dedupe**: `seen.json` (14-day memory) prevents re-judging URLs; title-token clustering merges multi-outlet coverage of one event and credits the extra outlets in the story footer.
- **Scaling up later**: when you outgrow flat JSON (archives, per-user features), swap `lib/store.js` for Supabase/Postgres — nothing else needs to change, all IO goes through that one file.
- **Copyright posture**: deks are written by Claude in its own words from titles/snippets and every story links out to the source. Keep it that way — the site is a router to sources, not a replacement for them.
